5. Key Dependency Summary

- users is the root table for almost all runtime relations.
- Strategy families:
  grid_strategies -> grid_orders
  dca_strategies -> dca_orders
  rebalancing_strategies -> rebalancing_assets, rebalancing_orders
- planned_orders is the internal order source of truth used by ORM/order manager/API paths.
- state_transition_logs is logically child-of planned_orders, but the live DB does not enforce that relation with a foreign key.
- sandbox_balances and sandbox_orders depend on users and support dry-run flows.
- user_sessions, bot_configs, backtest_results, audit_logs also depend on users.

6. Seed-Needed Tables

  Evidence:
  auth flow persists real users in users.
  create_user() always creates non-admin users.
  admin-only routes require an existing admin session.
  Live DB currently has at least 1 admin row and 4 non-guest rows.
  First-pass bootstrap reading:
  a fresh empty DB can accept guest sessions, but admin-only user/key management needs an admin user seeded or manually elevated.

11. Risk Notes

- Baseline migration built only from app/db/models.py will be incomplete and will not recreate the production schema.
- auth routes rely on users.login_fail_window_start and ip_login_failures; these are outside ORM-managed schema.
- current feature code heavily uses raw SQL for strategy/rebalancing/sandbox/session tables, so ORM metadata is not the real schema source of truth.
- state_transition_logs lacks a live FK to planned_orders, so orphaned transition rows are possible unless migration adds or intentionally preserves this gap.
- user_sessions and guest-user cleanup are DB-backed; dropping/restoring users without session coordination can create auth inconsistencies.
