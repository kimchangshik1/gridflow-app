GRIDFLOW RELEASE QUALITY GATES

1. 문서 목적

이 문서는 GridFlow를 외부 공개 또는 실납품하기 전에 내부에서 반드시 확인할 최소 통과 기준을 고정한다.
고객에게 제공하는 판매 문서가 아니다.
출시 가능 여부를 판단하기 위한 내부 품질 게이트 문서다.


2. 핵심 플로우 게이트

로그인:
- 정상 계정으로 로그인할 수 있어야 한다.
- 로그인 실패 시 원인이 화면 또는 로그에서 추적 가능해야 한다.

API 키 저장/반영:
- 사용자가 입력한 API 키가 저장되어야 한다.
- 저장된 키가 실제 런타임 설정에 반영되어야 한다.
- 키 저장 후 서비스 재시작 또는 재로그인 기준 동작이 문서와 일치해야 한다.

GUEST / DRY RUN / LIVE 구분:
- GUEST 사용자는 실거래 기능에 진입할 수 없어야 한다.
- DRY RUN은 실제 주문을 내지 않아야 한다.
- LIVE는 사용자별 저장 키로만 실거래해야 한다.
- 화면, 로그, DB 상태에서 현재 모드가 혼동되지 않아야 한다.

수동 주문 진입:
- 수동 주문 화면 또는 API 진입 경로가 정상 동작해야 한다.
- 주문 전 필수 입력값 검증이 동작해야 한다.
- 실패 시 silent failure 없이 실패 사유가 남아야 한다.

Grid / DCA / Rebalancing 기본 동작:
- 각 전략은 생성, 조회, 실행 상태 확인이 가능해야 한다.
- 기본 설정값으로 최소 1회 정상 실행 경로가 검증되어야 한다.
- 비정상 조건에서는 주문이 중단되고 원인이 기록되어야 한다.


3. 런타임 안정성 게이트

반복 에러/경고 허용 기준:
- 동일 에러가 반복 발생하면 HOLD 이상으로 판정한다.
- 운영자가 조치하지 않아도 누적되는 경고는 허용하지 않는다.
- 외부 공개 전에는 알려진 반복 에러의 원인과 조치 방안이 기록되어야 한다.

60초 주기 잡음 로그 허용 여부:
- 60초 주기 작업 로그는 정상 상태 확인에 필요한 수준만 허용한다.
- 매 주기마다 의미 없는 경고, 예외, 스택트레이스가 남으면 HOLD다.
- 로그가 정상 운영 판단을 방해하면 외부 판매 금지다.

silent failure 금지:
- 주문, 전략 실행, API 키 반영, 백업, 알림 실패는 조용히 실패하면 안 된다.
- 실패는 화면, 로그, 알림, DB 상태 중 최소 하나에서 확인 가능해야 한다.
- 실패 원인을 추적할 수 없는 기능은 PASS 처리하지 않는다.


4. 운영 게이트

백업:
- 운영 DB 백업 절차가 존재해야 한다.
- 백업 파일 생성 위치와 보관 기준이 확인되어야 한다.
- 백업 실패 시 운영자가 알 수 있어야 한다.

복구 검증:
- 백업 파일로 복구 검증을 수행할 수 있어야 한다.
- 복구 검증 결과가 기록되어야 한다.
- 복구가 검증되지 않은 백업은 PASS 근거로 사용할 수 없다.

logrotate:
- 주요 로그에 logrotate가 적용되어야 한다.
- 회전 후 서비스가 로그를 계속 기록해야 한다.
- 로그 파일이 무제한 증가하면 HOLD다.

health alert:
- 핵심 서비스 장애를 감지하는 health alert가 동작해야 한다.
- alert 실패 자체도 확인 가능해야 한다.
- 알림 경로와 점검 방법이 문서화되어야 한다.


5. 설치 재현 게이트

새 서버 기준 설치 재현:
- 새 서버에서 문서만 보고 설치를 재현할 수 있어야 한다.
- 숨은 수동 작업이 있으면 문서에 반영해야 한다.
- 설치 중 필요한 계정, 패키지, 환경 변수, 서비스 파일이 누락되면 HOLD다.

서비스 기동:
- 설치 후 필수 서비스가 정상 기동해야 한다.
- 재부팅 후에도 필요한 서비스가 자동 기동해야 한다.
- 서비스 상태 확인 명령이 문서와 일치해야 한다.
- 현재 운영 기준 필수 서비스는 gridflow-app.service, orderlens-ops.service, upbit-bot.service, nginx.service, postgresql@16-main.service다.
- upbit-web.service는 현재 운영 경로가 아니므로 문서나 점검에서 필수 active 대상으로 취급하지 않는다.
- 현재 endpoint 기준은 로컬 8000(/), 로컬 8010(/monitor), HTTPS /, HTTPS /monitor다.

문서와 실제 동작 일치:
- 설치 체크리스트의 명령과 실제 서버 동작이 일치해야 한다.
- 더 이상 쓰지 않는 명령, 파일명, 포트, 서비스명이 문서에 남아 있으면 HOLD다.
- 문서와 구현이 어긋나면 구현 또는 문서 중 하나를 수정한 뒤 재판정한다.


6. 보안/운영 기본 게이트

불필요 외부 포트 차단:
- 외부 노출 포트는 운영에 필요한 최소 범위여야 한다.
- DB, 내부 관리 포트, 개발용 포트는 외부에 열려 있으면 안 된다.
- 포트 노출 상태는 외부 공개 전 확인되어야 한다.

권한/키 노출 금지:
- API 키, DB URL, 세션 키, 토큰은 저장소와 로그에 노출되면 안 된다.
- env 파일과 키 파일은 필요한 사용자만 읽을 수 있어야 한다.
- 운영 로그에 민감 정보가 찍히면 FAIL이다.

공용 env 키 의존 금지 여부:
- LIVE 거래는 공용 env 키에 의존하면 안 된다.
- 사용자별 저장 키가 우선 적용되어야 한다.
- 공용 키가 필요한 경우 용도, 범위, 제거 계획을 기록해야 한다.


7. 문서 일치 게이트

scope / runbook / checklist와 실제 기능 일치:
- 책임 경계 문서의 범위와 실제 기능이 일치해야 한다.
- 인수인계 런북의 운영 절차가 현재 서버 상태와 일치해야 한다.
- 설치 체크리스트의 단계가 새 서버 기준으로 재현 가능해야 한다.
- 문서 간 충돌이 있으면 외부 공개 전 하나의 기준으로 정리해야 한다.


8. 판정 방식

PASS:
- 모든 필수 게이트를 통과했다.
- 남은 이슈가 외부 공개 또는 실납품에 영향을 주지 않는다.

HOLD:
- 핵심 기능은 동작하지만 외부 판매 전 해결해야 할 리스크가 남아 있다.
- HOLD 상태에서는 외부 판매 금지다.
- HOLD 항목은 원인, 영향, 담당자, 재판정 조건을 기록해야 한다.

FAIL:
- 핵심 플로우, 보안, 설치 재현, 운영 안정성 중 하나 이상이 명확히 깨졌다.
- FAIL 상태에서는 외부 공개, 데모, 실납품을 진행하지 않는다.
