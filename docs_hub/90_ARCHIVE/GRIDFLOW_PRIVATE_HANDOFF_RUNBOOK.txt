GridFlow 고객 인수인계 런북

1. 문서 목적

이 문서는 GridFlow 설치 후 고객이 직접 운영 상태를 확인하고, 기본 점검을 수행하기 위한 인수인계용 런북이다.
개발자를 위한 내부 구현 문서가 아니며, 코드 구조나 개발 방법을 설명하지 않는다.

고객은 이 문서를 기준으로 다음을 확인한다.

- 설치 후 시스템이 정상 접속되는지
- 로그인과 계정 상태가 맞는지
- 거래소/API 키 설정이 올바른지
- DRY RUN 또는 LIVE 모드가 의도한 상태인지
- 장애 발생 시 어디부터 확인해야 하는지


2. 최초 설치 후 확인 순서

설치가 끝난 뒤 고객은 아래 순서로 확인한다.

1. 접속 확인
   - 전달받은 도메인 또는 서버 주소로 접속한다.
   - 브라우저에서 화면이 열리는지 확인한다.
   - HTTPS가 적용된 경우 주소창에 보안 경고가 없는지 확인한다.

2. 로그인 확인
   - 전달받은 관리자 또는 고객 계정으로 로그인한다.
   - 로그인 실패 시 아이디, 비밀번호, 계정 활성 상태를 먼저 확인한다.

3. 거래소/API 키 설정 확인
   - 거래소 계정에서 발급한 API 키가 등록되어 있는지 확인한다.
   - 필요한 권한이 설정되어 있는지 확인한다.
   - IP 제한을 사용하는 경우 현재 서버 IP가 허용되어 있는지 확인한다.
   - API 키 원문은 별도 안전한 장소에 보관하고, 화면이나 문서에 노출하지 않는다.

4. DRY RUN/LIVE 상태 확인
   - 테스트 운용이면 DRY RUN 상태인지 확인한다.
   - 실거래 운용이면 LIVE 상태인지 확인한다.
   - 의도하지 않은 LIVE 실행을 막기 위해 운영 시작 전 모드를 반드시 확인한다.

5. 최근 로그/주문/활동 확인
   - 최근 주문 내역에 의도하지 않은 주문이 없는지 확인한다.
   - 최근 활동 로그에 반복 오류가 없는지 확인한다.
   - 봇 시작 직후 permission denied, invalid access key, rate limit, database error 등이 반복되는지 확인한다.


3. 서비스 실행/상태 확인

GridFlow 운영 시 기본적으로 아래 서비스가 정상 실행 중이어야 한다.
현재 운영 기준 서비스명은 아래 항목을 기준으로 확인한다.

- 앱 서비스
  - gridflow-app.service
  - 웹 화면과 API가 정상 응답해야 한다.

- 봇 서비스
  - upbit-bot.service
  - 자동매매 엔진, 주문 상태 확인, 정합성 체크가 실행된다.

- 운영/모니터링 서비스
  - orderlens-ops.service
  - 모니터링 화면 또는 운영 보조 기능이 정상 응답해야 한다.

- 제외된 구 웹 서비스
  - upbit-web.service는 현재 운영 경로가 아니다.
  - 서버에 남아 있어도 disabled/inactive 상태가 정상이다.

정상 예시:

- systemd 상태가 active (running)이다.
- gridflow-app.service, orderlens-ops.service, upbit-bot.service가 active 상태다.
- 웹 앱은 HTTPS / 경로에서 응답하고, 모니터는 HTTPS /monitor 경로에서 로그인 화면으로 이동한다.
- 최근 journal에 같은 오류가 계속 반복되지 않는다.
- 웹 화면 접속과 로그인에 문제가 없다.
- 주문, 전략, 활동 로그 화면이 열린다.

비정상 예시:

- 서비스 상태가 failed, inactive, activating 반복 상태다.
- 재시작 횟수가 계속 증가한다.
- permission denied가 반복된다.
- invalid_access_key가 반복된다.
- DB 연결 실패가 반복된다.
- 화면은 열리지만 주문/전략 데이터가 비어 있거나 오류가 표시된다.


4. 기본 운영 점검 루틴

매일 볼 것:

- 웹 화면 접속 가능 여부
- 로그인 가능 여부
- 주요 서비스가 active 상태인지
- 최근 활동 로그에 반복 오류가 없는지
- 주문이 의도한 모드와 계정에서 실행되고 있는지
- 서버 디스크 용량이 부족하지 않은지
- 백업 파일이 최근 날짜로 생성되고 있는지

주문 이상 시 볼 것:

- 해당 주문의 심볼, 가격, 수량, 상태
- DRY RUN 주문인지 LIVE 주문인지
- 거래소 API 응답 메시지
- API 키 권한과 IP 제한
- 최소 주문금액 미달 여부
- 잔고 부족 여부
- 거래소 점검 또는 rate limit 여부

로그 이상 시 볼 것:

- 같은 오류가 반복되는지
- 오류 발생 시각이 특정 작업과 연결되는지
- 서비스 재시작 직후에만 발생하는지
- 특정 심볼 또는 특정 사용자 계정에서만 발생하는지
- 최근 API 키, 서버, 전략 설정을 변경했는지


5. 장애 시 1차 점검 순서

장애가 발생하면 아래 순서로 확인한다.

1. 서비스 상태
   - 앱, 봇, 운영/모니터링 서비스가 active 상태인지 확인한다.
   - failed 또는 restart loop 상태라면 최근 journal을 확인한다.

2. 로그
   - 장애 발생 시각 전후의 journal을 확인한다.
   - permission denied, invalid_access_key, rate limit, database error, connection error를 우선 확인한다.

3. API 키
   - API 키가 삭제, 폐기, 만료되지 않았는지 확인한다.
   - 거래 권한이 켜져 있는지 확인한다.
   - IP 제한이 서버 IP와 맞는지 확인한다.
   - 최근 키를 교체했다면 시스템에도 같은 키가 반영되어 있는지 확인한다.

4. 거래소 응답
   - 거래소 점검 공지가 있는지 확인한다.
   - 거래소 API 장애가 있는지 확인한다.
   - 주문 거절 사유가 잔고, 권한, 최소금액, rate limit 중 무엇인지 확인한다.

5. 백업 상태
   - 최근 백업 파일이 있는지 확인한다.
   - 복구 검증이 최근에 성공했는지 확인한다.
   - DB나 서버 디스크 용량 부족이 없는지 확인한다.


6. 백업/복구 확인 포인트

백업 파일 존재:

- 백업 디렉터리에 최근 날짜의 DB 백업 파일이 있는지 확인한다.
- 백업 파일 크기가 비정상적으로 작지 않은지 확인한다.
- 백업이 오래 끊겨 있으면 즉시 원인을 확인한다.

복구 검증 의미:

- 복구 검증은 운영 DB를 덮어쓰는 작업이 아니다.
- 별도 임시 DB에 백업 파일을 복원해 데이터가 실제로 복구 가능한지 확인하는 절차다.
- 백업 파일이 존재하더라도 복구 검증에 실패하면 실제 장애 상황에서 사용할 수 없을 수 있다.

고객은 정기적으로 백업 생성 여부와 복구 검증 결과를 확인해야 한다.


7. 모드/계정 관련 확인

GUEST:

- 데모 또는 제한된 확인용 계정으로 본다.
- 실제 거래 계정으로 사용하지 않는다.
- 고객 운영 계정과 혼동하지 않는다.

DRY RUN:

- 실제 거래소 주문을 보내지 않는 테스트 모드다.
- 주문 ID가 내부 가짜 ID로 생성될 수 있다.
- 전략 흐름과 화면 동작을 확인하는 데 사용한다.

LIVE:

- 실제 거래소 API 키로 주문이 나갈 수 있는 운영 모드다.
- 실행 전 계정, API 키, 권한, IP 제한, 전략 설정을 다시 확인한다.
- 의도하지 않은 계정으로 LIVE 주문이 나가지 않도록 사용자와 키 매핑을 확인한다.

잘못된 계정/키 사용 방지:

- 고객별 API 키는 고객별 계정에만 연결한다.
- 공용 키를 임시로 넣어 여러 사용자 주문에 쓰지 않는다.
- 키 교체 후에는 작은 범위에서 상태조회와 주문 가능 여부를 검증한다.
- 사용하지 않는 키는 거래소에서 폐기한다.


8. 고객이 하면 안 되는 것

고객은 운영 안정성을 위해 다음 작업을 임의로 하지 않는다.

- 앱 코드 직접 수정
- 서버에서 검증되지 않은 패키지 설치
- systemd 서비스 파일 임의 변경
- nginx 설정 임의 변경
- DB 데이터를 직접 수정
- 백업 없이 서버 설정 변경
- API 키를 교체한 뒤 검증 없이 LIVE 운영
- 알 수 없는 오류가 반복되는데도 자동매매를 계속 실행
- 동일 계정에 여러 봇이나 외부 자동매매 도구를 동시에 연결

필요한 변경이 있으면 변경 목적, 대상, 예상 영향, 되돌리는 방법을 정리한 뒤 진행한다.


9. 지원 요청 전 고객이 정리해야 할 정보

지원 요청 전 아래 정보를 정리하면 원인 확인이 빨라진다.

- 발생 시각
  - 날짜와 시간을 가능한 정확히 기록한다.

- 심볼
  - 문제가 발생한 종목을 기록한다.
  - 예: KRW-BTC, KRW-ETH, KRW-XRP

- 화면/로그 증상
  - 화면에 표시된 오류
  - journal 또는 앱 로그의 오류 문구
  - 주문 상태와 전략 상태

- 최근 변경사항
  - API 키 변경 여부
  - 전략 설정 변경 여부
  - 서버 재시작 여부
  - 도메인, nginx, SSL, 방화벽 변경 여부
  - 거래소 계정 보안 설정 변경 여부

- 영향 범위
  - 전체 접속 장애인지
  - 특정 사용자만 문제인지
  - 특정 거래소 또는 특정 심볼만 문제인지
  - DRY RUN만 문제인지 LIVE만 문제인지


10. 문체 및 사용 기준

이 문서는 고객이 실제 운영 중 확인할 내용을 정리한 실무형 문서다.
과장된 표현이나 수익 보장 표현을 사용하지 않는다.
법률 문서처럼 해석하기 어렵게 쓰지 않고, 장애 상황에서 바로 확인할 수 있는 기준 문장으로 사용한다.

설치 완료 후 고객에게 이 문서를 전달하고, 고객이 직접 1차 점검을 수행할 수 있는지 확인한다.
운영 대행, 야간 대응, 장애 분석, 서버 복구 지원이 필요한 경우에는 별도 유지보수 범위로 협의한다.
