GridFlow 설치형 납품 범위 및 운영 책임 경계

1. 상품 성격

GridFlow는 고객 서버에 설치해 전달하는 설치형 자동매매 운영 시스템이다.
본 상품은 고객 환경에 시스템을 구축하고 초기 동작을 확인한 뒤 인수인계하는 납품형 상품이다.

GridFlow 납품은 운영 대행 상품이 아니다.
납품 이후의 일상 모니터링, 매매 판단, 중지 판단, 계정 관리, 서버 운영 책임은 고객에게 있다.


2. 포함 범위

다음 항목은 기본 납품 범위에 포함된다.

- 고객 서버 또는 합의된 서버 환경에 GridFlow 설치
- 서비스 실행에 필요한 기본 환경 설정
- 도메인, HTTPS, nginx, systemd 등 기본 운영 구성 확인
- 앱, 봇, 모니터링 화면의 기본 기동 검수
- DB 백업, 복구 검증, 로그 로테이션 등 기본 운영 장치 구성
- 운영 런북 전달
- 설치 결과, 주요 명령, 점검 방법 인수인계

초기 설정은 납품 시 합의된 범위 안에서 수행한다.
고객별 전략 설계, 장기 운영 판단, 실시간 대응 체계는 별도 계약이 없는 한 포함하지 않는다.


3. 미포함 범위

다음 항목은 기본 납품 범위에 포함되지 않는다.

- 24시간 모니터링
- 수익 보장
- 실시간 운영 대행
- 매수, 매도, 중지, 재개에 대한 지속적인 판단 대행
- 거래소 정책 변경에 대한 상시 대응
- 외부 API 변경에 대한 상시 대응
- 서버 장애 발생 시 상시 복구 대행
- 고객 계정, 고객 서버, 고객 네트워크 문제에 대한 지속 운영 책임
- 고객이 임의로 변경한 설정, 코드, 서버 환경에 대한 무상 복구

필요한 경우 위 항목은 별도 유상 운영 지원 범위로 협의한다.


4. 무상 하자보수 범위

무상 하자보수는 납품 범위 안에서 확인되는 하자에 한한다.

무상 하자보수 대상은 다음과 같다.

- 전달 문서와 다르게 동작하는 명확한 버그
- 납품 범위에 포함되어 있었으나 누락된 설치 항목
- 초기 설정 오류로 인해 기본 서비스가 정상 기동하지 않는 문제
- 납품 당시 합의한 기능 범위 안에서 발생한 재현 가능한 오류

무상 하자보수는 투자 결과, 거래소 응답, 시장 상황, 고객 운영 판단의 결과를 보정하는 것을 의미하지 않는다.


5. 유상 지원 범위

다음 항목은 기본 납품 이후 유상 지원 범위로 본다.

- 신규 기능 추가
- 기존 기능의 동작 방식 변경
- 매매 전략 추가 또는 전략 로직 변경
- 고객 요청에 따른 UI, 문구, 화면 구조 변경
- 서버 운영 지원
- 장애 원인 분석
- 로그 분석 및 데이터 정리
- 거래소 API 변경 대응
- 야간, 주말, 공휴일 대응
- 고객이 직접 변경한 서버, 코드, 설정에 대한 복구

유상 지원은 요청 내용, 긴급도, 작업 범위, 대응 시간에 따라 별도 견적 또는 유지보수 계약으로 진행한다.


6. 고객 책임 범위

납품 이후 고객은 다음 항목을 직접 관리해야 한다.

- 서버 비용 결제 및 서버 계정 관리
- 도메인, DNS, SSL 갱신 관련 계정 관리
- 거래소 계정 보안 관리
- 거래소 API 키 발급, 권한 설정, 폐기, 재발급 관리
- API 키 유출 방지
- 시스템 운영 모니터링
- 자동매매 실행, 중지, 재개 판단
- 전략 설정값 검토
- 투자금 규모 결정
- 거래소 공지, 정책 변경, 점검 일정 확인
- 백업 파일 보관 상태 확인

고객은 자동매매 시스템이 항상 정상 수익을 보장하지 않는다는 점을 이해하고 운영해야 한다.


7. 장애 시 1차 점검 항목

장애 또는 이상 동작이 의심될 때 고객은 먼저 다음 항목을 확인한다.

- 서비스 상태
  - systemd 서비스가 active 상태인지 확인
  - 앱, 봇, 모니터링 서비스가 재시작 루프에 빠져 있지 않은지 확인

- 로그
  - 최근 journal 로그 확인
  - 앱 로그와 모니터링 로그 확인
  - permission denied, invalid access key, rate limit, database error 등 반복 오류 확인

- API 키
  - 거래소 API 키가 만료, 폐기, 삭제되지 않았는지 확인
  - 필요한 권한이 부여되어 있는지 확인
  - IP 제한이 현재 서버 IP와 맞는지 확인

- 거래소 응답
  - 거래소 점검 여부 확인
  - 거래소 API 장애 또는 rate limit 여부 확인
  - 주문 거절 사유 확인

- 백업/복구 확인
  - 최근 DB 백업 파일 존재 여부 확인
  - 복구 검증 절차가 문서대로 가능한지 확인
  - 서버 디스크 용량 부족 여부 확인

위 항목 확인 후에도 원인이 불명확하면 로그와 발생 시각을 정리해 지원을 요청한다.


8. 면책 성격 문구

GridFlow는 자동매매 실행을 보조하는 시스템이며 투자 성과를 보장하지 않는다.
시장의 가격 변동, 유동성 부족, 슬리피지, 거래소 주문 거절, 거래소 장애로 인한 손실은 고객 책임 범위에 속한다.

다음 사유로 발생한 손실, 장애, 중단은 기본 납품자의 지속 운영 책임에 포함되지 않는다.

- 투자 손실
- 거래소 장애
- 거래소 API 장애
- 거래소 정책 또는 API 사양 변경
- 외부 API 응답 지연 또는 제한
- 고객 서버 장애
- 고객 서버 비용 미납
- 고객 계정 탈취 또는 API 키 유출
- 고객이 직접 변경한 설정, 코드, 네트워크, 방화벽 문제
- 고객의 실행, 중지, 재개 판단 지연

다만 납품 범위 안의 명확한 설치 하자나 문서와 다른 동작은 무상 하자보수 기준에 따라 처리한다.


9. 전달물 목록

기본 납품 시 전달물은 다음과 같다.

- 설치 완료된 GridFlow 시스템
- 서비스 기동 상태 확인 결과
- 운영 런북
- 기본 점검 체크리스트
- DB 백업 절차
- DB 복구 검증 절차
- 로그 위치와 로그 로테이션 설명
- 장애 시 1차 점검 방법
- 주요 systemd 서비스명과 재시작 명령
- 고객 인수인계 설명

전달물은 고객이 직접 운영 상태를 확인하고 1차 점검을 수행할 수 있도록 하기 위한 기준 자료다.


10. 문서 사용 기준

이 문서는 계약서 자체가 아니라 GridFlow 설치형 납품의 실무 범위와 운영 책임 경계를 설명하기 위한 기준 문서다.
견적, 계약, 유지보수 조건은 별도 문서에서 금액, 기간, 대응 시간, 작업 범위를 명시해 확정한다.

고객에게 납품 전 이 문서를 공유하고, 기본 납품 범위와 운영 대행 범위가 다르다는 점을 확인한 뒤 진행한다.
